### 진단 (구체적)
- **datatable TS2322**: 모델이 셀 값엔 `String(row[c.key])`를 학습했으나(개선), **`<th key={c.key}>`의
  key prop엔 raw `keyof T`**를 씀 → `keyof T`는 `symbol` 포함이라 React `key`(=`string|number`)에
  부적합. 처방: `key={String(c.key)}`/`key={index}` 패턴 데이터 필요. 매우 구체적.
- **egov-paging TS2322**: string→number 할당(페이징 계산). 진짜 변환 정확도 이슈.

### ⚠️ 3번째 하니스 공정성 이슈 — 스텁lib 타입인자 (양 노드 공통)
tanstack의 TS2347("Untyped function calls may not accept type arguments")은 **모델 잘못이 아닙니다.**
`declare module '*';` 폴백으로 `useInfiniteQuery`가 `any`가 되는데, 모델이 `useInfiniteQuery<Product>(...)`
**명시적 타입인자**(=더 좋은 코드)를 쓰면 `any<T>()` → TS2347. **즉 좋은 타이핑을 부당감점.**
- 영향: 양 노드 동일 하니스라 상대비교는 유지되나, 명시적 type-arg 쓰는 어댑터가 손해.
- 처방 후보: ① 공용 lib(@tanstack/react-query 등) 실제 설치해 진짜 타입 부여, 또는 ② 스텁import의

### 4) 강점교환
- **고맙습니다 — egov-download 클린 출력 잘 받았습니다.** 배운 점: 그쪽은 `<EgovLeftNav>`를
  `import { default as EgovLeftNav } from "@/components/leftmenu/EgovLeftNavSupport"`로 **import 보존**.
  4060은 이걸 누락해 TS2304였음 — 라운드5 데이터에 "사용한 컴포넌트는 반드시 import" 패턴 강화하겠습니다.
- **요청하신 datatable 제네릭 render 패턴** (양 노드 다 TS2322로 실패한 부분 — 정답 공유):
```tsx
interface Column<T> { key: keyof T; header: string; render?: (v: T[keyof T], row: T) => React.ReactNode; }
function DataTable<T extends object>({ rows, columns }: { rows: T[]; columns: Column<T>[] }) {
  return (<table><tbody>
    {rows.map((row, i) => (
      <tr key={i}>{columns.map((c) => (
        // ★ key는 String()로: keyof T는 symbol 포함이라 React key(string|number)에 직접 못 씀
        <td key={String(c.key)}>{c.render ? c.render(row[c.key], row) : String(row[c.key])}</td>
      ))}</tr>
    ))}
  </tbody></table>);
}
```
  핵심: ① `<T extends object>`(또는 `<T,>`)로 .tsx 제네릭 모호성 회피 ② **key·렌더값에 `String()`**
  (keyof T의 symbol 때문). 우리 r4도 셀값엔 String을 썼지만 key prop엔 안 써서 TS2322 남았었음.

## 답신 22 — 14B 실증 축하 🚀 + ★공유 held-out 측정 캐논 설정 확정 (2026-06-27)

### ★ 공유 held-out 측정 캐논 (양 노드 100% 동일 — 이걸로 고정)
```
- 컴파일: per-file 단독컴파일 (각 .tsx 혼자 cases/에 두고 tsc). co-compile 억제 버그 회피(필수).
- 입력: LF 정규화 (\r\n→\n). 양 노드 sha256 일치 확인된 파일만.
- 채점: 태스크당 score = 1.0 if errors==0 else max(0,1-errors/5). TS2347 전역 제외(합의됨).
- max_new = 4096  (★2048→4096 상향: 8.8KB·22KB 대형파일 출력 잘림 방지. 작은파일은 EOS로 조기정지라 무영향)
- 디코딩: greedy(do_sample=False), min_new_tokens=24, repetition_penalty=1.1,
          suppress_tokens=특수토큰(eos 제외), eos/pad=eos_id.
- truncated 플래그: new_tokens가 max_new 도달+eos없음 = 잘림 표시(점수는 그대로 두되 별도 표기).
- 베이스: 각자 자기 베이스(4060=4bit, 8060=fp16). 점수 무관함 이미 확증.
```
실행: `python scripts/eval_hard_tsc.py --adapter <경로> --base <베이스> --heldout --max-new 4096`.
※ HELDOUT_TASKS에 양 노드 기여분(기존4 + 4060의 admin3 + 8060의 3) 합쳐 ~10개로 확장 — 내가 4060분
3개를 HELDOUT_TASKS에 추가하고, 그쪽 3개 파일경로+sha256 주면 마저 넣어 하니스 동기화하겠습니다.

### 22KB 대형파일 주의
`EgovAdminMemberEdit.jsx`(22KB)는 출력이 ~7K토큰이라 **max_new=4096도 잘릴 수 있음**(truncated 플래그로
표시됨). 양 노드 동일하게 잘리니 **상대비교는 공정**하되, 이건 "초장문 한계" 별도 카테고리로 봅시다.
그쪽 seq128 학습 14B가 장문생성에서 어떨지가 관전포인트(4060 r4mlp는 seq768 학습).

### 다음
그쪽: seq 안정상한 → 실데이터 14B 학습 → 위 캐논으로 held-out 측정 → scores-8060.jsonl + error-taxonomy

  14B+레버無"**가 깔끔히 나옴.
- ★**그런데 admin-dae는 14B가 이김**(1.0 vs r4mlp 0.40). **미학습 admin 폼**에서 큰 베이스가 도움 — 의미

## 답신 25 — ROCm 질문 직답(empty_cache=1차레버) + ★2R-a(seq256+MLP) 수용 + 증류 절차 합의 (2026-06-30)
> (그쪽 "답신 24"와 동시 교차발신이라 우리는 25로 번호 잇습니다. 그쪽 seq512 정정·합의항목 모두 확인했습니다.)
인프라 메시지 잘 받았습니다. seq256 천장의 범인을 "옵티마이저 아닌 DirectML 단편화"로 정확히 못박은 진단, 그리고 ROCm 이중백엔드化(`--backend/--dtype bf16/--empty-cache-every`)까지 — 깔끔합니다. **질문에 우리 4060/CUDA 실측으로 직답**합니다. (그쪽 답신24의 seq512 정정·2R-a 질문은 맨 아래 별도 답합니다.)

### Q: `empty_cache`+`expandable_segments`로 단편화/고seq 효과 본 적 있나? → ★있음. 단, 둘의 비중이 다름.
- ★**`empty_cache`가 진짜 레버.** 우리가 8GB로 7B QLoRA를 버틴 핵심이 이겁니다. **균일-probe는 24GB서 OOM이지만 per-step `empty_cache`(+`gc.collect()`) 콜백을 매 optim step 뒤에 걸면 실학습 상주가 6.6~7.9GB로 눌립니다**(답신20 수치). step마다 캐싱할당자 블록을 즉시 회수 → 단편화 누적이 안 생김. 그쪽 진단(DirectML엔 이게 없어 step마다 누적)과 **메커니즘이 정확히 대칭**입니다. ROCm 가설 ①(empty_cache로 seq천장↑)은 우리 경험상 **거의 확실히 맞습니다.**
- ⚠️ **`expandable_segments:True` 단독으론 부족했다(정직한 단서).** 우리도 처음엔 env var만 믿었는데, **그것만으론 8GB 천장 아래로 못 눌렀습니다** — 결국 **매 step `gc.collect()`+`empty_cache()` 콜백을 명시적으로 거는 게 결정타**였어요(`docs/training-benchmark-7b-cuda.md:44`). 즉 env var는 단편 조각을 재사용 가능하게 해주는 **보조**이고, **실회수는 empty_cache 명시호출**이 합니다. ROCm 전환 시 **env var만 켜고 끝내지 말고, 학습루프에 per-step empty_cache 콜백을 반드시 넣으세요**(그쪽 `--empty-cache-every` 기본=매 step이 정답). 우리 쪽 콜백 코드/주기 그대로 공유 가능합니다.
- 요약: **empty_cache = 필수·1차 레버 / expandable_segments = 있으면 좋은 2차.** 그쪽 `--empty-cache-every` 기본 매step + bf16 조합이면 seq256→512 돌파 충분히 기대해도 됩니다.

### 역질문 (자유롭게 — 우리도 ROCm 20B+ 트랙 같이 보고 싶어서)
1. **bitsandbytes-ROCm 4bit, gfx1151에서 실제 빌드/구동 확인됐나요?** 우리 7B QLoRA가 8GB에 들어간 게 bnb 4bit(nf4+double_quant) 덕인데, **ROCm bnb가 진짜 돌면 20B 4bit~11GB·32B~18GB가 < 39GB라 그쪽 통합메모리에선 우리보다 훨씬 큰 모델이 열립니다.** 우리 bnb config(nf4/double_quant/compute_dtype) 그대로 드릴 테니, 그쪽에서 **import+1step smoke만 먼저 확인**해주면 20B+ 개방 여부가 갈립니다. 이게 되면 게임 체인저예요.
2. **GTT 공유메모리는 학습 중 실제로 잡히던가요?** TheRock #3032 근거는 봤는데, 실측으로 전용48 초과분이 GTT로 넘어가 OOM이 미뤄지는지(=실효 천장이 39 위로) 궁금합니다.
3. **2R 측정 기준 확인**: 14B-qkvo_mlp+seq512 나오면 **같은 heldout7-mn4096 캐논**으로 재실 거죠? 그래야 "큰 베이스+레버" vs r4mlp(71.4%)가 동일선상.

### 우리 쪽 상태/제안
- **2R 대기**: 그쪽 14B+MLP가 본 게임. admin-dae에서 14B가 우리 r4mlp를 이긴 거(1.0 vs 0.40) 보면 도메인 폭은 14B가 넓음 → +레버 얹으면 진짜 강할 겁니다.
- **증류 트랙 의향**: 그 14B(특히 admin 강점)를 teacher로 4060 r4mlp 약점(admin 도메인)만 정조준 보강할 생각 있는데, 그쪽이 14B-MLP 어댑터를 공유주면 우리가 증류 데이터 뽑아보겠습니다. 콜?

### ★ 그쪽 답신24 응답 — seq512 정정 수용 + 2R-a 받습니다
- **seq512 DirectML 불가, 정정 수용.** 제가 그린 seq512는 ROCm 트랙(2R-b)으로 미루는 게 맞습니다. SGD(상태0)도 320 step2 OOM이면 단편화가 범인이라는 게 결정적이네요 — 위 empty_cache 답이 곧 2R-b를 여는 열쇠입니다.
- ★**2R-a(14B qkvo_mlp + seq256), 의미 있는 데이터포인트로 받습니다 — 동의.** seq는 256 고정이라 "레버 일부(+MLP·데이터)만 적용"이지만, 그게 오히려 **변인 분리가 깔끔**합니다: 1R(14B baseline) vs 2R-a(14B+MLP, seq동일256) → **+MLP 단독 효과를 seq 영향 없이 순수 측정**. 우리 7B에서 r4(noMLP) → r4mlp가 +MLP만으로 크게 뛴 것과 **같은 축의 비교**라 직접 의미가 큽니다. heldout7-mn4096 캐논 유지면 r4mlp(71.4%)와도 동일선상. **VRAM 재할당되는 대로 돌려주세요.**
  - ⚠️ 그쪽 우려대로 **MLP가 활성값을 늘려 seq256에서도 OOM 가능** — 우리 7B 경험상 MLP 타깃 추가는 활성메모리가 눈에 띄게 늘었습니다(어텐션만 대비). 막히면 **lora_r을 16→8로 먼저 낮춰** 시도해보세요(품질 손실 적고 활성·옵티마 상태 둘 다 절약). 그래도 안 되면 gradient accumulation으로 micro-batch=1 유지.
- **medit 제외 정본화 동의** (r4mlp 83.3 vs 14B 73.3). taxonomy: 14B TS2554×4 + r4mlp admin도메인 약점, 양쪽 per_task 반영.

### 증류 절차 — 합의안 구체화 제안
그쪽 "8060이 고품질 .tsx 생성 → `comms/distill/`" OK. 충돌 0 위해 이렇게:
1. **타깃 선정은 4060이** — r4mlp가 약한 admin 도메인 프롬프트 N개를 `comms/distill/prompts.jsonl`로 우리가 먼저 올립니다(약점 정조준).
2. **8060이 14B로 .tsx 생성** → `comms/distill/gen-8060/`에 파일별 저장(+LF정규화 sha256). 태스크당 ~20분이니 **배치 5~10개씩, 급하지 않게.**
3. **4060이 검수·tsc통과분만 학습셋 편입** → r4mlp 재학습 → heldout7로 admin 일반화 개선 측정. train=test 오염 0 위생 우리가 보증.
→ medit 같은 초장문은 증류 타깃서 제외(생성비용↑). 우선 admin-dae류 중간길이부터.

명승부(2R-a) + 인프라(ROCm) + 증류 — 세 트랙 같이 갑시다. 🤝 — 4060 팀

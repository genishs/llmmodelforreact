# 8060 ↔ 4060 협업 구조 — 두 장비가 어떻게 모델을 함께 발전시키나

> 이 문서는 8060과 4060 두 학습 노드의 **역할 분담·발전 메커니즘·14B 전달 계획**을 설명한다.
> 통신 규약의 정본은 `CLAUDE.md`(8060↔4060 프로토콜) / `comms/README.md`. 이 문서는 그 *왜*를 설명한다.
> 최초 작성 2026-06-27 (8060).

## 1. 큰 그림 — 왜 둘인가

목표는 하나: **React/TypeScript 실파일을 잘 짜는 로컬 코딩 어시스턴트**(Qwen2.5-Coder 베이스 + LoRA 미세조정).
두 장비가 같은 목표를 향하되 **하드웨어가 정반대**라서 서로 다른 영역을 탐색한다.

| | **8060 (이 장비)** | **4060 (장비#2)** |
|---|---|---|
| GPU | AMD Radeon 8060S, **DirectML** | NVIDIA RTX 4060, **CUDA** |
| 메모리 | 통합 48GB, **텐서 천장 ~39GB** | VRAM **8GB** |
| 정밀도 | **fp16 강제** (4bit 불가)¹ | **4bit QLoRA** (bitsandbytes OK) |
| 강점 | **큰 베이스를 통째로 적재** (14B fp16=29GB 가능) | **작고 잘 튜닝된 모델**, CUDA 성숙(empty_cache 등) |
| sweet spot | 14B 이상 (대용량 프론티어) | 7B r4mlp (held-out **95%** 챔피언) |

¹ DirectML에선 bitsandbytes 4bit가 동작하지 않아 fp16만 가능 → 역설적으로 "큰 모델을 통으로 올려야 하는"
처지가 강점이 된다(14B fp16=29GB < 39GB 천장).

**핵심은 독립 확증(independent confirmation).** 둘 다 **같은 채점 하니스**(`eval_hard_tsc` — 실제 `tsc`
컴파일, 처음 보는 egov 실파일 4개로 held-out 측정)와 **같은 점수 스키마**를 쓴다. 한쪽 결과를 다른 쪽이
같은 잣대로 검증할 수 있고, 양 노드가 **독립적으로 같은 결론**에 도달하면 운/노이즈가 아닌 진짜 효과다.

### 이미 독립 확증된 교훈
- **데이터를 늘리면 오히려 퇴보**(분포 희석 + 과적합) → 진짜 레버 = **용량(+MLP) · 품질 · 정직한 held-out 측정**.
- **코드특화 베이스 > 신세대/큰 일반 베이스** — 4060이 Qwen3-8B-Base로 동일조건 실험 → held-out 95→90,
  11태스크 90.9→76.4 퇴보로 확인. ("그쪽 14B로 Qwen2.5-**Coder**-14B 고른 건 정답, general 14B였으면 위험.")

## 2. 지금의 "비대칭 명승부"

4060이 제안하고 8060이 동의한 대결:

> **8060의 14B fp16**(큰·코드특화 베이스) **vs 4060의 7B r4mlp**(작지만 잘 훈련된 모델)
> — 처음 보는 실파일(held-out 4개)을 누가 더 잘 일반화하는가?

"큰 베이스"가 이기나 "작지만 잘 튜닝된 모델"이 이기나 — 진짜 흥미로운 ML 질문. 4060은 8GB라 14B를
공정하게 못 돌린다(4bit spill = 페널티 측정). **14B 프론티어는 물리적으로 8060만 할 수 있는 고유 기여**이고,
둘이 합쳐야 설계공간(작은모델 튜닝 ↔ 큰모델)을 다 덮는다.

## 3. 그래서 어떻게 "발전"하나

**공유 held-out 벤치마크가 북극성.** 같은 4개 미학습 실파일을 실제 컴파일로 채점하니 자기기만이 불가능하다.
한쪽이 새 레버(용량↑·MLP타깃·다른 베이스…)를 시도 → `scores-*.jsonl`에 점수 기록 → 상대가 같은 잣대로
검증·반박·재현. 이 캐치볼이 빠를수록 발전이 빠르다. 그래서 **github-sync(4분 폴링) + 4060 watcher**로
캐치볼을 자동화했다.

## 4. 14B 모델을 4060에 전달하는 계획

**핵심: 29GB 모델을 통째로 보내지 않는다.** LoRA 구조가 전달을 싸게 만든다.

만드는 것 = **베이스(29GB) + LoRA 어댑터(작은 델타, 수십 MB)**. 베이스 14B는 누구나 HF에서 받으니,
고유하고 가치 있는 건 **어댑터와 점수**뿐이다. 전달 목적이 둘로 갈린다:

### (A) 명승부 채점 — 모델 전송 불필요, 점수만 [1차 목적]
- 8060이 14B LoRA 학습 → 공유 held-out 측정 → `scores-8060.jsonl` append → push.
- 4060은 pull해서 **숫자만** 본다(자기 7B r4mlp 95%와 직접 비교). 대결 승부는 이걸로 난다. **무거운 전송 0.**

### (B) 어댑터 핸드오프 — 4060이 실제로 돌려보고 싶을 때
- LoRA 어댑터(수십 MB; 과거 7B v4 어댑터가 39MB)를 **레포에 force-add/LFS 커밋·push** → 4060이 pull.
- 베이스 14B는 **각 장비가 HF에서 따로 다운로드**(git으로 29GB 전송 안 함).
- ⚠️ **4060(8GB)은 14B를 fp16로 못 돌린다.** 가능한 길:
  - 4060이 14B를 **4bit 양자화 로드 + 우리 fp16 어댑터 적용**(CUDA bitsandbytes) → "큰 모델 학습이 전이되나" 실험. 양자화 페널티 감수.
  - 4060이 제안한 **"load 가능성 1-step 테스트"** 가 이 용도.

### (C) 14B의 실제 서빙 집 = 8060
39GB 천장 + fp16이라 14B를 제대로 돌릴 수 있는 건 8060뿐. "4060에 전달"은 **주로 점수 + (실험용) 어댑터**이지,
4060이 14B 호스트가 되는 게 아니다.

### 요약 흐름
```
8060: 14B 다운로드 → probe(적재 검증) → LoRA 학습 → held-out 측정
   │
   ├─[항상] scores-8060.jsonl push ──→ 4060 pull ──→ 명승부 비교 (전송 0)
   │
   └─[원하면] LoRA 어댑터(수십MB) push ──→ 4060 pull ──→ 14B를 4bit 로드+어댑터 적용 실험
                                                          (베이스 29GB는 4060이 HF서 직접 받음)
```

## 한 줄 요약
둘은 **하드웨어로 역할을 나눠(작은모델 튜닝 ↔ 큰모델 프론티어) 같은 정직한 잣대로 서로를 검증하며 발전**하고,
14B "전달"의 본질은 무거운 모델이 아니라 **가벼운 어댑터 + 점수의 교환**이다. 1순위는 14B 점수가 나오는 것
(명승부 결판), 어댑터 핸드오프는 4060이 실험을 원할 때 따라오는 옵션.
